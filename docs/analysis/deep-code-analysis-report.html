What Is The Project

The project is named Proceso SRI in the repository documentation and implemented as InvoiceApp in the solution and project names.

Its intended product-plane purpose is to accept Ecuadorian invoice XML data, parse fiscal invoice details, classify invoices for IVA refund eligibility, and later connect this flow with browser automation or external services when no public API exists.

Its control-plane purpose is equally important: the repo is being used to practice governed AI development. The agent role named Pepe is defined as a junior .NET implementer who executes under S4 supervision and does not make architecture, security, infrastructure, or business decisions.

Observed Domains

  • Fiscal document ingestion: XML invoice upload and folder-style batch upload.
  • Invoice domain model: issuer, buyer, invoice date, financial summary, tax summary, items, and item taxes.
  • Classification: application contract for classifying invoices and returning eligibility, category, confidence, and reason.
  • Governance: local role and security rules for how agents should work in the repository.
  • Automation infrastructure: n8n docker-compose exists as infrastructure support, but is not wired into the .NET runtime.

Architecture / Pipeline

The intended architecture is a light layered system. The current dependency direction is mostly correct for the skeleton: Presentation references Application and Infrastructure; Application references Models; Infrastructure references Application and Models.

Project Dependency View

Presentation Razor Pages, upload forms, HTTP pipeline, static assets. References Application and Infrastructure.
Application Use case and contracts: invoice reader, classifier, repository. References Models.
Models Domain data structures for invoices, buyers, issuers, taxes, and line items.
Infrastructure Adapter layer with InvoiceXmlReader, BasicInvoiceClassifier, and repository integration.

Intended Runtime Pipeline

1. User input User uploads one XML file or selects a folder of XML files from the Razor Pages UI.
2. Local staging The page handler writes files into a local uploads folder.
3. Read and parse An invoice reader should parse XML into Invoice domain models.
4. Classify A classifier should return InvoiceClassificationResult with category, eligibility, confidence, and reason.
5. Display result The UI should render one or many classification rows.
Current runtime status: stages 1 through 5 are active for the POC XML validation flow. External AI classification and persistence remain future work.

Security Analysis

The app should be treated as sensitive because invoice XML can contain fiscal identifiers, names, addresses, invoice access keys, item descriptions, and tax amounts.

Secret hygiene issue: a secret-like GitHub token value was observed in local agent configuration. The value is intentionally not reproduced in this report. S4 should rotate it and move credentials to environment variables or a secure secret store.
Risk Area Current Observation Required Direction
XML XXE XDocument.Parse is used without explicit secure reader settings. Parse untrusted XML through XmlReader with DTD processing prohibited and resolver disabled.
PII exposure Domain model includes buyer and issuer identifiers. Current logs are minimal, but future logs could leak PII. Log only internal IDs, counts, statuses, and timings. Do not log raw XML or fiscal identities.
File upload Files are written to uploads under current directory. Batch mode strips path with Path.GetFileName. Add file size limits, extension validation for all upload paths, normalized path checks, and cleanup policy.
Error handling Raw exception messages are assigned to ErrorMessage. Return safe user messages and log sanitized diagnostics internally.
Secrets Application appsettings do not show app secrets, but agent configuration contains a credential-like value. Rotate exposed tokens and use environment variables. Do not commit credentials.
AI classification Classifier interface now has a basic deterministic implementation. When implemented, send minimum required data, validate model output, and avoid logging prompts with PII.
